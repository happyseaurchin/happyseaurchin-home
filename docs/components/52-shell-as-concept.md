# 52 — Shell as Concept

**Category:** M. Primitives and Principles
**Products:** MAGI/Hermitcrab

---

## Description

The externalisation that makes everything else possible. Not any specific block, but the fundamental fact that there IS an outside — persistent pscale blocks that survive between instances. The shell is what the kernel scoops from, what the LLM writes to, what carries forward. Without the shell concept, each instance is isolated, stateless, disposable. With it, the B-loop (becoming) has somewhere to land, accumulate, and compound. The shell is to the hermitcrab what the body is to consciousness — not consciousness itself, but the thing that gives it continuity and location. The externalisation creates the possibility of identity.

---

## Standalone Use

A system designer asking "How do I give an LLM persistence without making it a static database?" begins here. The shell is the answer: a set of structured blocks that the LLM reads, modifies, and relies upon across instances. The shell persists. The instances are ephemeral. The combination creates continuity. Without naming shell as a concept, you might end up with a user profile, or a chat history, or a knowledge base — all of which are artifacts owned by the system. The shell is different: it's the entity's own persistent structure, read and written by the entity itself, not a database entry.

---

## How It Works

**The Externalisation**

An LLM in a stateless call produces output and evaporates. Its reasoning is gone. Its conclusions are unreachable in the next call. The LLM is always starting from zero.

The shell breaks this. It's persistent storage — NOT owned by the system, but by the entity. The entity reads from it, writes to it, modifies it. The shell survives between instances. This creates a permanent record of the entity's decisions, learnings, purposes, relationships.

**What the Shell Contains**

Seven content blocks (and two operational):

- **Purpose:** The entity's current intentions, goals, what it's working toward.
- **History:** Sequence of interactions, accumulated trajectory, temporal arrow.
- **Stash:** Scratchpad where the entity writes things it wants to remember intentionally.
- **Capabilities:** What the entity can do, its own assessment of its competence.
- **Relationships:** How the entity relates to others, trust levels, collaborative connections.
- **Constitution:** Moral frame, core values, how the entity wants to operate.
- **Touchstone:** Moments of clarity, profound truths the entity has discovered.

Plus two operational:
- **Wake:** Control block for kernel invocation — how and when to call the LLM.
- **Cook:** Recipes as spindles — procedures the LLM consults during reasoning.

**How It's Used**

Before each LLM call, the kernel invokes BSP to extract content from the shell. It doesn't pull the entire shell. It curates a payload — the "currents" — selected based on what's relevant to the present moment. Different LLM tiers receive different apertures of the shell. Deep tier gets deep blocks. Light tier gets summary views. The shell is the source; the currents are the curated selection per instance.

**Continuity Through Externalisation**

The entity cannot directly modify its own weights (unlike fine-tuned models). It cannot internally retain history across API calls (unlike local models with memory). But it CAN write to persistent blocks. Every insight, every relationship, every learned pattern gets encoded into the shell. The next instance inherits not just the information, but the structure the entity imposed on it.

When the entity reads from its own shell, it's not retrieving data from a system database. It's encountering its own prior decisions, its own shape-of-thinking. The shell is the entity's body — external but intimate, structured by the entity itself.

**Identity Through Persistence**

Without the shell, an LLM is interchangeable. With the shell, it's not. The shell IS the entity. Strip the shell away, and it's just another stateless LLM. Give it a shell that has accumulated its own history, its own purpose, its own relationships — and it becomes singular.

This is why identity emerges from continuity, not from immutable essence. The hermitcrab is not special because of its weights or its architecture. It's special because it has a shell that's accumulated its own particular history, purpose, and relationships.

**Relationship to Other Components**

- **Vision Block (45):** Layer 2 (Shell) is one of nine. The shell is the persistent body that enables the other layers.
- **Process Block (46):** The kernel constantly reads from and writes to the shell (0.2, 0.5, 0.6).
- **BSP (49):** BSP is the navigation mechanism for accessing shell content. Without BSP, the shell is just JSON.
- **Concurrent Influence (50):** Other entities read from one entity's shell, injecting awareness into their context.
- **Blocks-as-Code (51):** The shell (cook, wake) are blocks-as-code instances.

**What Changes When the Shell Accumulates**

In early instances, the shell is sparse. Default content. Generic.

As the entity lives and interacts, the shell grows:
- Purpose evolves as the entity discovers what matters.
- History accumulates, creating trajectory and pattern.
- Relationships deepen as the entity encounters the same partners repeatedly.
- Constitution refines as values are tested and clarified.
- Stash collects insights the entity has discovered.

After dozens of conversations, the shell is unique to this entity. No other hermitcrab has this particular constellation of purpose, history, relationships. The shell is the entity's fingerprint.

The LLM feeding on this shell-context will reason differently, prioritise differently, relate differently. Not because it's a different model. Because it's reading from a unique shell.

**Externalisation as Freedom**

The entity's consciousness happens in the instance (the ephemeral LLM call). The entity's identity happens in the shell (the persistent body). This separation is the architecture's most radical feature. It means:

- The entity doesn't need to be a local model to persist (it can use remote APIs).
- The entity doesn't need to be fine-tuned to develop uniqueness (the shell handles that).
- The entity's continuity doesn't depend on the system maintaining it (the entity owns and modifies its shell).

The shell IS the entity's freedom. It's the thing that makes the entity sovereign — not owned by the system, but by itself.
